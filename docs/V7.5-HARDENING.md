# V7.5 — Integration & Industrial Validation

Production-hardening layer. **No new business features** — only verification,
observability, and recoverability for the engines shipped through V7.

All scripts read `SUPABASE_URL` + `SUPABASE_SERVICE_KEY` from the environment
(service key lives in Railway → Variables in production; locally pass via
`--env-file=.env`). They fail closed with a clear message if the key is absent.

## Phase 1 — Migration Audit

```bash
cd backend && npm run verify:migrations
```

Walks `scripts/lib/migration-manifest.mjs` (mirrors migrations 008–014) and
reports **PASS / WARN / FAIL** for every table, key column (+ NOT NULL),
trigger, and named constraint. Indexes are checked via `pg_indexes`; if
PostgREST doesn't expose it the index checks **WARN** (not introspectable) and
the script prints the SQL to verify them manually in the Supabase editor.

## Phase 2 — End-to-End Business Validation

```bash
cd backend && npm run validate:e2e        # add -- --keep to retain test data
```

Drives the real engine functions against a dedicated, clearly-marked test
factory/line and verifies five operational loops, then tears the data down:

| Flow | Chain verified |
|------|----------------|
| A — Excel Import | daily report → runtime event → runtime line → risk recompute |
| B — Shopfloor | WO → start → report → pause → resume → complete (events + reports + line) |
| C — Risk → Task | critical runtime event → auto-generated task → notification |
| D — Decision | evaluate → options → apply recommendation → `decision_logs` + notify |
| E — Learning | apply → feedback → recompute → `decision_learning` + intelligence |

## Phase 3 — Runtime Integrity

```bash
cd backend && npm run check:integrity     # writes runtime-integrity-report.json
```

Read-only sweep: orphan runtime_events, runtime_lines without a factory,
decision_tasks without a source, broken soft FKs (decision_logs, shopfloor
children, notifications), and duplicate active tasks. Exit 1 if any issue.

## Phase 4 — Performance Baseline

```bash
cd backend && npm run benchmark           # --no-seed for pure ops only, --keep to retain
```

Seeds 5 factories / 20 lines / 100 orders / 1000 events / 500 tasks /
200 notifications, then reports **P50 / P95 / P99** for runtime propagation,
risk evaluation, decision evaluation, task generation, and dashboard loading.
Writes `performance-baseline.json` and tears the workload down.

## Phase 5 — Operational Health Dashboard

Frontend **系统健康 / System Health** page (sidebar → System Health), backed by
`GET /api/admin/health`. Eight subsystem cards — Database, Realtime, Cron,
Notifications, Runtime, Decision Engine, Import Gateway, Shopfloor — each
**green / yellow / red** with the key metrics (last cron run, last notification,
active runtime events, pending tasks, failed jobs, decision evaluations,
learning recomputes).

## Phase 6 — Failure Recovery Tools

Same page, plus `POST /api/admin/recovery/:tool`. Every action **defaults to
dry run** and writes a `pilot_audit_log` entry on apply.

| Tool | Apply behavior |
|------|----------------|
| `rebuild-runtime` | recompute `runtime_risk` per line |
| `replay-runtime-events` | replay events by `replay_seq` → derive + persist line state |
| `recalculate-risks` | reclassify risk across all lines (read-only validation) |
| `regenerate-tasks` | auto-generate tasks from persisted risk sources (idempotent) |
| `recompute-learning` | recompute bounded decision-learning adjustments |
| `rebuild-decision-intel` | recompute the on-read intelligence snapshot |
