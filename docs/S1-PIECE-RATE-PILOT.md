# S1 计件试算 — 单线现场试点清单

目标:在**一条真实产线**上验证计件试算,与组长手记对账,判断是否可推广。
**全程试算并行,不发薪、不生成应付工资记录。**

---

## 一、上线前准备(Day 0)

### 1. 跑迁移 015(验证)
- 在 Supabase SQL Editor 执行 `supabase/migrations/015_piece_rates.sql`。
- 验证:`select * from piece_rates limit 1;` 不报错;打开「计件试算」页不报 500。

### 2. 配工序工价
- 「计件试算」页 → 工序工价表,录入试点线全部工序工价(如 平车 0.8、锁眼 0.3)。
- 工序名要和工单 `operation` 一致(系统已自动归一化:全角→半角、去空格、英文小写,所以"平 车"="平车")。

### 3. 选试点线
- 选 1 条配合度高的线;页面顶部「产线」选它。

### 4. 工人身份(关键)
- 每个工人扫码报工时 `reported_by` 必须是**稳定一致的工号或姓名**。
- 系统会归一化(去空格/全角),但**不能解决"同一个人今天写张三、明天写小张"**——必须现场统一成工号。
- 验收看板的「缺工人(条)」就是抓这个。

### 5. 选工单
- 确认试点线今天的工单都有正确的 `operation` 和 `line_id`。

---

## 二、生产当天(报工流程)

1. 工人用「车间执行台」扫码找到工单 → 报工,填**件数**,确认 `reported_by`。
2. 组长**照旧手工记产量**(这是对账基准,不能省)。
3. 全天系统自动:件数 × 工价 = 计件金额。

---

## 三、收工对账(Nightly)

打开「计件试算」页,选当天日期 + 试点线,看顶部看板:

| 卡片 | 含义 | 处理 |
|---|---|---|
| 报工件数 / 计件金额 | 系统试算总数 | 与组长手记总数比 |
| 缺工价(条) | 有报工没工价,按 0 计 | 立刻补工价 |
| 缺工人(条) | 报工没署名,无法归人 | 现场强调署名 |
| 疑似重复(条) | 同工单+同人+同件数 2 分钟内重复 | 核实是否双击/重扫 |

然后点「**导出对账表**」→ 生成 CSV(工人/系统件数/系统金额/手工件数/手工金额/差异),
交组长把手工数填进去,逐人对差异。

> 后端也有 `POST /api/payroll/piece-wages/reconcile`,可直接传手工数得到逐人差异和最大误差%。

---

## 四、判定是否可推广(验收)

连续记录每天:**总件数差异、总金额差异、逐人最大误差%、脏数据条数**。

- ✅ 可推广:逐人误差稳定 **<1%**,脏数据(缺工价/缺工人/重复)≈0,连续达标。
- ⛔ 暂不推广:误差大或脏数据多 → 先解决"工人不署名/工序对不上/重复扫"再说。

**误差<1% 且跑满一个完整发薪周期**前,绝不切正式发薪。

---

## 五、红线

- 本系统只做**试算**,不写应付工资、不标记"工资已定"。
- 正式发薪由工厂现有流程负责,直到对账连续达标、老板书面确认。
